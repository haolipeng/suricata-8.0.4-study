# COTP 分片重组问题详解

> 本文档面向初学者，详细解释 S1 问题（COTP 分片重组缺失）的本质、影响和修复方案。

---

## 一、协议栈的层次关系

IEC 61850 MMS 的数据在 TCP 上传输时，会被层层封装：

```
┌─────────────────────────────────────────────────────┐
│                    TCP 流（字节流）                    │
│  ┌───────────────────────────────────────────────┐  │
│  │              TPKT 帧（信封）                    │  │
│  │  ┌─────────────────────────────────────────┐  │  │
│  │  │         COTP 帧（快递单）                 │  │  │
│  │  │  ┌───────────────────────────────────┐  │  │  │
│  │  │  │      MMS PDU（真正的信件内容）      │  │  │  │
│  │  │  └───────────────────────────────────┘  │  │  │
│  │  └─────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

类比：
- **TPKT** = 信封，上面写了"这个信封总共多大"（length 字段）
- **COTP** = 快递单，上面写了"这是第几箱"以及"**是不是最后一箱**"（EOT 标志）
- **MMS PDU** = 信件内容，就是你真正要读的东西

## 二、当 MMS PDU 很小时——一切正常

比如一个简单的 Read Request，只需要读一个变量，数据很小（几十字节）：

```
TCP 流中只有一个 TPKT 帧：

┌────────────��─────────────────────────────┐
│ TPKT 头 (4B)                             │
│ version=3, length=50                     │
├──────────────────────────────────────────┤
│ COTP 头 (3B)                             │
│ type=DT, EOT=1  ◄── "这是最后一箱"       │
│                     (也是唯一一箱)         │
├──────────────────────────────────────────┤
│ MMS PDU (43B)                            │
│ ConfirmedRequest { Read, invokeID=1 }    │
│ ......完整的 PDU 数据......               │
└──────────────────────────────────────────┘

代码处理过程：
  parse_tpkt_cotp_frame → 拿到完整 MMS 载荷
  parse_mms_pdu → 成功解析 ✅
```

## 三、当 MMS PDU 很大时——问题出现

现在想象一个 Read Response，服务端返回了变电站里 500 个数据点的值。
MMS PDU 有 3000 字节，但一个 TPKT 帧最大只装得下约 2000 字节的载荷。
怎么办？**拆成多个 COTP 帧发送**：

```
TCP 流中有两个 TPKT 帧：

帧 1:
┌──────────────────────────────────────────┐
│ TPKT 头 (4B)                             │
│ version=3, length=2004                   │
├──────────────────────────────────────────┤
│ COTP 头 (3B)                             │
│ type=DT, EOT=0  ◄── "还没完,后面还有箱子" │
├──────────────────────────────────────────┤
│ MMS PDU 片段 1 (1997B)                   │
│ [A1 82 0B B8 02 01 01 A4 82 ...]        │
│ ← 这不是完整的 PDU！只是前半段            │
└──────────────────────────────────────────┘

帧 2:
┌──────────────────────────────────────────┐
│ TPKT 头 (4B)                             │
│ version=3, length=1010                   │
├──────────────────────────────────────────┤
│ COTP 头 (3B)                             │
│ type=DT, EOT=1  ◄── "这是最后一箱了"      │
├──────────────────────────────────────────┤
│ MMS PDU 片段 2 (1003B)                   │
│ [... 83 01 00 A5 03 ...]                 │
│ ← 这是后半段                             │
└──────────────────────────────────────────┘

完整 MMS PDU = 片段 1 + 片段 2 = 3000B
```

关键信号就是 COTP 头里的 **EOT（End of TSDU）标志**：

```
EOT = 0  →  "我只是其中一个分片，后面还有"
EOT = 1  →  "我是最后一个分片（或者唯一的一个）"
```

## 四、当前代码怎么处理的（错误的）

看 `mms.rs:260-333` 的核心循环：

```
while !start.is_empty() {
    parse_tpkt_cotp_frame(start)  →  拿到一个帧

    如果是 COTP DT 帧 且 有载荷:
        直接把载荷丢给 parse_mms_pdu()   ◄── 问题在这里！
}
```

图示：

```
帧 1 到达（EOT=0，只是片段 1）：
  ┌─────────────────┐
  │ MMS 片段 1      │──→ parse_mms_pdu()
  │ (不完整的数据)   │         │
  └─────────────────┘         ▼
                         BER 解码失败！长度说有 3000B，
                         但只给了 1997B
                              │
                              ▼
                     触发 MalformedData 事件 ❌
                     （实际上数据没问题，只是还没收全）

帧 2 到达（EOT=1，片段 2）：
  ┌─────────────────┐
  │ MMS 片段 2      │──→ parse_mms_pdu()
  │ (不完整的数据)   │         │
  └─────────────────┘         ▼
                         BER 解码失败！片段 2 的开头
                         不是合法的 TLV 标签
                              │
                              ▼
                     又触发 MalformedData 事件 ❌
```

**结果：一条完全正常的 MMS 响应，产生了两个错误事件，零条正确日志。**

## 五、正确的做法应该是什么

需要一个**重组缓冲区**，把分片攒齐了再解析：

```
帧 1 到达（EOT=0）：
  ┌─────────────────┐
  │ MMS 片段 1      │──→ 追加到重组缓冲区，先不解析
  └─────────────────┘    ┌──────────────────────┐
                         │ 缓冲区: [片段 1]      │
                         └──────────────────────┘

帧 2 到达（EOT=1）：
  ┌─────────────────┐
  │ MMS 片段 2      │──→ 追加到重组缓冲区
  └─────────────────┘    ┌──────────────────────┐
                         │ 缓冲区: [片段1][片段2] │
                         └──────────────────────┘
                                    │
                         EOT=1，说明收全了！
                                    │
                                    ▼
                         parse_mms_pdu(完整数据)
                                    │
                                    ▼
                         成功解析 ✅
                         清空缓冲区
```

## 六、这个问题在真实场景中有多常见

```
场景                               是否会触发分片    频率
─────────────────────────────────────────────────────────
读取单个数据点                      否               -
读取整个逻辑节点(LN)的所有属性       可能             中
读取整个逻辑设备(LD)的数据集         很可能           高
GetNameList 返回大量变量名           很可能           高
文件传输 (ObtainFile/FileRead)      必定             高
GOOSE 控制块配置的批量读取           可能             中
```

变电站、风电场等工业现场，上电初始化时客户端（HMI/SCADA）通常会批量读取所有数据模型，
这时几乎必然触发 COTP 分片。
也就是说，**当前代码在每次设备上电初始化时就会产生大量误报**。
